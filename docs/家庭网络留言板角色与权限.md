# 家庭网络留言板：角色与权限（设计 v1）

## 1. 背景与目标

当前实现使用 **guest / host 两把密码**：连上后可见**全部留言板**，仅权限档位不同。实际使用中存在问题：

- 客人来家里时，可能只应看到「旅行记录」等**指定留言板**，而不是整台服务器上的所有板。
- 同一家人（如你本人）和客人都应能看**同一块板**（一板多角色），而不是「板只属于一个角色」。
- AI Agent 的模型、工具、上下文预算不应全局一套，而应随**角色**变化。
- 手机侧不需要复刻服务器上的多角色体系；**本机**与**远程连接**两种身份即可（见 §5）。

本文定义 **PC 服务器侧** 的角色模型与 API 行为。实现状态：设计稿（待开发）；当前代码仍为 guest/host，迁移见 §7。

## 2. 核心概念

| 概念 | 说明 |
|------|------|
| **角色（role）** | 静态配置于 `config.json`，每个角色有独立密码、权限、Agent 能力 |
| **管理员（admin）** | 一种角色；**始终可见并管理全部留言板**，不依赖板的 `role_ids` 配置 |
| **普通角色** | 只能看到 **`role_ids` 中包含本角色**的留言板，只能在该板**发帖**（及读附件），无管理功能 |
| **留言板可见角色** | 一块板对应 **`role_ids: string[]`**，仅列出**非 admin** 角色；admin **不必写入**该列表 |

**一板多角色**是基本场景：例如 `travel-2025` 的 `role_ids: ["guest", "family"]`（**不含 admin**），则主人（admin 密码）与客人、家人均可见、均可发言；`family-private` 仅 `["family"]`，客人看不到。admin 无需出现在 `role_ids` 中。

### 2.1 admin 不变式（硬规则）

- **admin 角色始终可见全部留言板**，访问判定时不检查 `role_ids`。
- **`role_ids` 只描述非 admin 角色的授权**；写入或展示时**不得要求包含 `"admin"`**，服务端也会忽略误写入的 `"admin"`。
- 这样即使 `role_ids` 配错或漏配，admin 仍能进入、修改 `role_ids` 或**删除**该板，不会出现「谁都删不掉」的孤儿板。
- 新建板若省略 `role_ids`，默认为 `[]`：**仅 admin 可见**，再由 admin 分配给 guest 等角色。

**旧板兼容**：`meta.json` **无** `role_ids` 字段的留言板，视为对所有非 admin 角色可见（与早期 guest 看全板行为一致）。**显式** `role_ids: []` 表示仅 admin。

密码 → 角色映射在**服务端**完成；客户端每次请求携带 `X-Network-Service-Password`，服务端解析为 `role_id`，不在客户端保存角色表。

## 3. 服务器配置（静态）

```json
{
  "roles": {
    "admin": {
      "password": "…",
      "can_create_boards": true,
      "can_manage_boards": true,
      "can_assign_board_roles": true,
      "agent": {
        "models": [
          { "name": "qwen2.5", "context_chars": 24000 },
          { "name": "gpt-oss:latest", "context_chars": 12000 }
        ],
        "tools": {
          "attachments": true,
          "web_fetch": true
        }
      }
    },
    "guest": {
      "password": "…",
      "can_create_boards": false,
      "can_manage_boards": false,
      "agent": {
        "models": [],
        "tools": {}
      }
    },
    "family": {
      "password": "…",
      "can_create_boards": false,
      "can_manage_boards": false,
      "agent": {
        "models": [{ "name": "qwen2.5", "context_chars": 24000 }],
        "tools": { "attachments": true, "web_fetch": false }
      }
    }
  }
}
```

说明：

- **`password`**：连入该设备时使用的接入密码（可多个角色各一把；同一密码不得映射多个角色）。
- **`can_create_boards`**：是否允许 `POST /boards`。
- **`can_manage_boards`**：是否允许删除留言板、修改/删除留言、分配 `role_ids`、导入留言板包等。
- **`agent.models`**：该角色在 `@` 补全和 `GET /agent` 中可见、且 Agent 会为该板响应的模型列表；每项可配 **`context_chars`**（发给 LLM 的留言上下文预算，按**被 @ 的模型**取用，见 §6）。
- **`agent.tools`**：该角色下 Agent 可调用的工具子集（附件列表/读取、网络等）。

兼容迁移：过渡期可将旧 `guest_password` / `host_password` 映射为两个固定角色（见 §7）。

## 4. 留言板元数据

`meta.json` 在 v0 字段基础上增加：

```json
{
  "id": "travel-2025",
  "name": "2025 旅行",
  "revision": 12,
  "role_ids": ["guest", "family"],
  "created_at": 1710000000000
}
```

- **`role_ids`**：有权访问此板的**非 admin** 角色 ID 列表；**admin 始终可见，不必也不应写入此字段**（见 §2.1）。
- 创建板时（admin）：请求体带 `name` 与 `role_ids`；省略时默认 `[]`（仅 admin，待分配）。

## 5. API 行为摘要

认证头不变：`X-Network-Service-Password`。

| 接口 | admin | 普通角色（且在板的 role_ids 中） | 普通角色（不在列表中） |
|------|-------|-----------------------------------|------------------------|
| `GET /boards` | 全部板（或仅列出，管理入口显示 role_ids） | 仅授权板 | 不出现该板 |
| `GET /boards/{id}/messages` | 允许 | 允许 | 404 / forbidden |
| `POST …/messages` | 允许 | 允许 | forbidden |
| `POST /boards` | 允许 | forbidden | forbidden |
| `DELETE /boards/{id}` | 允许 | forbidden | forbidden |
| `PUT /boards/{id}`（改 name、role_ids） | 允许 | forbidden | forbidden |
| `PUT/DELETE …/messages/{id}` | 允许 | forbidden | forbidden |
| `POST /boards/import` | 允许（见导入导出文档） | forbidden | forbidden |
| `GET /agent` | 返回本角色 agent 配置 | 同左 | 同左 |

响应字段扩展：

- `GET /boards` 每项含 `role_ids`（admin 可见；普通角色可省略或只读）。
- 连入会话可返回 **`role_id`**、**`role_label`**（可选展示名），供 UI 显示「当前身份：客人」。
- `GET …/messages` 中 `agents` / `commands` / `participants` 按**当前角色**过滤（与现有一致，但 agents 来自角色配置而非全服）。

内部命令（`/ai status`、`/ai stop`）是否对非 admin 开放：建议 **凡该角色配置了 Agent 即可用**，与发帖权限一致。

## 6. AI Agent 与上下文

- Agent **进程仍随 PC 服务运行**；某板被 @ 时，校验该板 `role_ids` 是否包含触发者角色，且该角色是否允许被 @ 的模型。
- **模型列表、工具、上下文预算均按角色配置**，不再使用全服单一 `agent.models` / `max_board_context_chars`。
- **上下文截断**（已实现逻辑，待改为按模型预算）：
  - 从**最新留言**向旧取，**整条留言**为最小单位，不截断单条中间；
  - 超出当前 **@ 模型** 的 `context_chars` 时省略更早留言，并加说明前缀；
  - 单条留言长于预算时仍**整段保留**；
  - `ai_status` 状态行**不进入** LLM 上下文。

## 7. 与当前实现的差异（迁移）

| 当前 | 目标 |
|------|------|
| `guest_password` / `host_password` | `roles.{id}.password` |
| 所有板对 guest 可见 | `GET /boards` 按 `role_id` ∩ `board.role_ids` 过滤 |
| `agent` 全服一段 | 每角色 `agent.models` / `tools` |
| `max_board_context_chars` 全局 | 每模型 `context_chars` |
| 手机远程 host = 改删一切 | 远程 **admin 密码** = 管理员角色；guest 等 = 普通角色 |

建议步骤：① 配置与解析 role；② 板 `role_ids` + 列表过滤；③ Android 远程 UI 按返回的 `role_id` 隐藏管理项；④ Agent 按角色；⑤ 按模型上下文预算。

## 8. 手机侧：本机 vs 远程

手机**不**在本地实现完整多角色表：

| 模式 | 身份 | 能力 |
|------|------|------|
| **本机留言板**（`isSelf`） | 本机管理员 | 管理本机所有本地板：创建、删除、发帖、附件、导出/导入包（见导入导出文档） |
| **远程连接** | 由所填密码在**服务端**解析的角色 | 仅服务端允许的操作；UI 显示解析到的 `role_id`（如「远程 · 客人」） |

远程连接时，若密码对应 **admin**，则与 PC 上管理员等价：可建板、改 `role_ids`、删板、改删消息。非 admin 仅能在可见板内发帖。

本机服务可不跑 Agent（与现况一致）；Agent 仅在 PC `bulletin_server.py`。
