# 家庭网络共享策略

## 介绍

本文构想一个家庭网络共享的方案。我们使用手机，常常需要一家人（也可以理解为一个
共同的组织）一起共享一组信息。网络运营商常常在公网上提供这种服务。但很多人并不
能接受信息直接被公开到网络上。

所以，我想着用家庭（或者组织）的局域网来完成这种信息共享，基本思路是：在家庭的
PC和手机上运行一个服务，注册mDNS服务端口，这样，当我们进入局域网的时候，就通过
这个服务发现互相的服务端口，我们可以连上这些用户，然后可以给它发信息，这些信息
可以是对话，文件，目录等内容。一旦我们认知了一个用户，这个用户就算离线我们也可
以给他发信息，但这些信息会留在队列中，等待下次连上了，再一次推送过去。

手机不使用运行服务（费电），所以我们是在本地管家中会实现成ftp那个功能一样：打开
tab才会上线，离开tab就离线。所以，这个是基础功能，我们的重点在这里：我们可以在
家庭网络中放置一些服务器角色，比如我们可以在Mac Mini上运行一个服务器，这个服务器
也是一个通讯方，它就是长期在线的，我们也可以给它发消息，而它的工作是，把信息分类
备份下来，作为这个家庭（或者组织）的信息中心，然后我们设计一个AI Agent，专门管
理和总结这些信息，我可以开发一组工具，不像那些通用Agent那么功能强大，只能用来在
这个有限的信息管理空间中做做总结，查查网络相关材料，快速查询关键信息，等等。这样，
这些服务器角色就可以用来提供“家庭秘书”的服务了。我希望反复调优这个Agent，让它能
充分利用一些能在Mac Mini上运行的低智模型，也能完成一些基本的信息管理工作，它做
复杂的推理肯定不够，但根据需要管理信息，配合工具构造的规则，不让它破坏消息，应
该是可以做到的。比如我们可以用程序逻辑来保证备份确切发生到固定的位置，然后AI只能
用特定位置的信息进行检索，并生成特定位置的“信息视图”，这样，信息就隔离开了。

这种组织服务器可以不止一个，你可以在家里是一个服务器，到社区俱乐部是另一个服务
器，到一个小公司是另一个服务器。在同一个局域网也可以有生活组，工作组，旅游组等
不同的群，每个都是独立的信息隔间。在隔间中，我们可以部署不同级别的Agent，高度保
密的用本地模型，无所谓的用公共模型，从而让AI的能力可以和个人日常生活和工作联系
在一起。

## AI的Idea评审意见

我先把Idea给AI做了一个初步的讨论，结论和我设想基本一致，我先记录一下，后面落实
设计的时候再删除：

```

从 家庭网络共享策略.md 这个想法本身看，我认为没有致命漏洞，方向是成立的，而且和
这个项目“本地优先、隐私优先、文件与知识管理结合”的气质是匹配的。它不是一个伪需
求，也不是技术上明显走不通的路线。真正的问题不在于“能不能做”，而在于你一开始把
它理解成什么系统。如果你把它定义成“家庭局域网里的去中心化通信与知识中枢”，它是
可行的；如果你把它定义成“没有中心节点也能像公网 IM 一样稳定离线投递的全对等系
统”，那就会很快撞墙。

**核心判断**

mDNS 适合作为发现机制，不适合作为信任机制，也不适合作为离线能力基础。它只能回答
“附近现在有什么服务”。离线投递要想可靠，几乎必然需要一个长期在线的存储转发节点。
你文里的 Mac Mini 不是锦上添花，而是最关键的基础设施。手机上“打开页面才上线”不
是缺点，反而是一个很清晰的产品边界。它天然节电，也避免了 Android 常驻后台带来的
复杂性。代价只是：手机不能承担可靠收件箱角色。这个系统更像“局域网内的私有信息空
间”，不是“聊天软件”。如果你用前者来定义它，很多设计都会变得更合理。高层设计上最
需要守住的边界

把发现、通信、存储、AI 四层分开。mDNS 只负责发现；通信协议只负责传输；中心节点
负责存储转发和归档；AI 只读经过授权的数据视图。不要混在一起。把“用户”与“设备”分
开。人是身份主体，手机、PC、家庭服务器只是设备。否则以后多设备、多群组、多服务
器漫游时会很乱。把“组织空间”定义成一等概念。你文里家庭、公司、俱乐部、生活组、
工作组，本质上都是不同的信息隔间。这个概念很强，建议正式化，不要只把它当成一个
标签。把“消息”定义成事件流，而不是一堆可随意修改的对象。只要有离线、补投、多端
同步、AI 摘要，事件流模型会比传统“改数据库里某条记录”更稳。这个 Idea 里真正的结
构性风险

最大风险不是技术，而是角色混淆。你现在的描述里同时有“节点”“联系人”“服务器”“群组”
“AI Agent”，这些概念还没完全分层，后面很容易职责重叠。第二个风险是安全目标不够
明确。你既希望隐私，又希望服务器备份、检索、总结。如果服务器要做 AI 总结，它通
常就得看到明文；如果你想端到端加密，AI 就只能看授权后的副本或摘要。这里必须尽早
选边界。第三个风险是把“离线消息”想得太像公网 IM。家庭局域网的价值不一定是高实时
聊天，可能更像“共享收件箱、共享资料、共享任务、共享提醒、共享知识卡片”。如果你
产品定义得更偏后者，系统会更稳定，也更符合本地网络特性。第四个风险是多服务器漫
游。家里一个、公司一个、社区一个，这个想法很美，但它会立刻引入身份映射、权限迁
移、数据隔离、策略切换。这个方向可以有，但应该是后续能力，不该进第一阶段的定义。
从方案选择上，我更推荐的路线

首选是“中心节点优先”的家庭内网模型。所有短时在线设备通过发现找到家庭服务器，再
和它做收发与同步。这是最稳的。不建议第一阶段做真正的设备间全互联 mesh。它对发现、
连通性、状态同步、补投、冲突处理的要求都更高，但用户价值未必更高。如果你后面要
做跨组织、多群组、多 AI 的扩展，最好把服务器理解成“一个信息域”，而不是“一个普通
联系人”。这样未来的家庭、公司、俱乐部本质上只是多个域。Git 风格同步适合资料归档、
共享目录、版本留痕，不太适合高频消息主通道。你现有项目里的 .sysgit 更适合承担知
识库、共享文件、配置快照、公钥分发，不建议直接演化成聊天总线。这个系统最有价值
的地方，其实不是聊天如果只做通信，它会和很多现有工具重叠。但如果你把它做成“家庭
私有信息基础设施”，就很有特色。自然能长出来的功能包括：

家庭共享收件箱。谁上传了什么、谁留了什么消息、谁要谁处理什么事。家庭知识库。路
由器配置、宽带账号、家电维护记录、老人用药说明、孩子学习资料，都能归档并检索。
家庭任务与值班。谁去拿快递、谁处理缴费、谁接送、谁维修。设备目录与能力表。哪台
机器能打印、哪台能存档、哪台能跑模型、哪台能做网关。自动化秘书。先别把它想成“万
能 Agent”，更像“基于固定数据区做整理、提醒、摘要、问答”的受限助手。安全审计与时
间线。谁在什么时候上传、阅读、确认、删除了什么。情境化群组。生活组、工作组、旅
游组这些，不只是聊天群，而是完整的隔离空间。有没有更好的替代思路有，但不是“更
好”，而是“更适合不同目标”。

如果你的目标是强实时聊天，那成熟公网协议思路会更省事，但这会偏离你的本地私有目
标。如果你的目标是家庭资料与消息归档，当前这个本地网络中心节点思路其实比公网 IM
更贴切。如果你的目标是跨网络、跨地点稳定使用，那最终还是要引入可穿透的入口或公
网中继，否则局域网方案天然受场景限制。如果你的目标是 AI 助手，最好的基础不是“更
复杂的聊天”，而是“更规整的结构化事件和资料仓库”。我会给出的高层结论这个 Idea 值
得做，方向上没有明显错误，而且有产品辨识度。最好的高层修正不是推翻，而是重新定
义它：不要把它当“一个局域网聊天功能”，而要把它当“一个局域网内的私有信息域平台”，
通信只是其中一种事件类型。这样一来，mDNS、中心节点、文件共享、知识归档、AI 总结、
群组隔离，这些部件就都能自然放进同一个架构里。
