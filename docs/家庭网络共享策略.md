# 家庭网络共享策略

## 1. 介绍

本文构想一个家庭网络共享的方案。我们使用手机，常常需要一家人（也可以理解为一个
共同的组织）一起共享一组信息。网络运营商常常在公网上提供这种服务。但很多人并不
能接受信息直接被公开到网络上。

所以，我想着用家庭（或者组织）的局域网来完成这种信息共享，基本思路是：在家庭的
PC和手机上运行一个服务，注册mDNS服务端口，这样，当我们进入局域网的时候，就通过
这个服务发现互相的服务端口，我们可以连上这些用户，然后可以给它发信息，这些信息
可以是对话，文件，目录等内容。

### 1.1 设备与留言板

我们**不**在全网定义「用户 ID」，而是：

- 任何一台设备都可以提供**留言板服务**（PC 常驻服务器，手机可选开本地服务）。
- 每台设备上可有**多块留言板**；连上对方设备后，先看到**你有权访问的板列表**，再
  进入某一块板发言。
- 你在某块板上的显示名（`author_label`）由发帖时填写，**只在板内有效**。

### 1.2 角色与密码（目标模型）

早期实现使用 **guest / host 两把密码**，连上后可见**全部留言板**，权限仅两档。这
对「客人只能看旅行记录、不能看家庭私密板」等场景不够用。

**目标模型**（详见 [家庭网络留言板角色与权限](./家庭网络留言板角色与权限.md)）：

- 在 **PC 服务器** `config.json` 中**静态配置多个角色**，每角色独立密码。
- 连入时密码 → **角色**；`GET /boards` 只返回该角色有权访问的板。
- 每块板配置 **`role_ids: []`**（**一板多角色**，仅列非 admin 角色）：例如旅行板授权 `guest`，主人用 admin 密码**无需写入列表**即可见；私密板不含 `guest`。
- **admin 始终可见全部板**（硬规则，不依赖 `role_ids`），避免配置错误导致无人可删板。
- **管理员角色**可创建/删除板、分配 `role_ids`、改删消息、导入导出包；其他角色仅
  能在授权板内**发帖**（及读附件）。
- **AI Agent** 的模型、工具、上下文预算按**角色**配置，而非全服一套。

手机侧不复制完整角色表，只区分：

- **本机留言板**：本机管理员，管理本地所有板（含导出/导入包）。
- **远程连接**：密码在服务器解析角色；admin 密码等价远程管理员，其余为受限角色。

### 1.3 本地板与服务器归档

旅途中可在手机**仅本地**建板记录；回家后通过 **留言板包（boardpack）** 导入 PC
服务器长期维护，或导出备份后删除本机板。格式与流程见
[家庭网络留言板导入导出](./家庭网络留言板导入导出.md)。

手机侧本地服务仍可「开了界面才上线、退出离线」；本地板也可关闭，仅作远程客户端。

### 1.4 典型场景：旅行记录

1. **外出**：在手机「本机留言板」新建一块板，纯本地记录（可不开启网络服务）。
2. **回家**：将该板**导出**为 `.boardpack`，**上传到 PC 服务器**（admin 导入），并设置
   `role_ids: ["guest"]`（admin 无需列入）。
3. **待客**：客人用 guest 密码远程连接，只能看到含 `guest` 的板（含旅行板），主人用
   admin 密码同样可见——**一板多角色**，不会出现「客人能看、主人反而不能看」。
4. **腾空间**：上传成功后可选**删除本机板**；需要时再从包文件或服务器**导入**回来。

## 2. 设计思路

### 2.1 留言板的存储管理

留言板的内容保存在应用的私有存储中（Android）或配置的 `board_root`（PC），每个留言
板一个独立目录，包括 `meta.json`、`messages.json` 与 `attachments/`（见
[家庭网络附件协议](./家庭网络附件协议.md)）。

客户端连上后同步消息列表；附件按需下载。不预设存储上限（可信环境内先用满再议）。

每个设备放一个默认留言板，便于初始化。

**当前代码**仍支持 guest/host 双密码；**目标**为 per-role 密码与板级 `role_ids`，见
角色文档 §7 迁移说明。

### 2.2 diff协议

消息序列可以是一个带序列号的单条消息的列表，如果消息是一个附件，可以带上特殊标
记说明附件的信息即可。

设备第一次连上的时候先同步一次全集，之后如果主列表发生更新，就针对这个连接加一
条“增删改”的记录，定期同步到客户端即可。

如果设备发生离线，就只能看到离线前的消息（并且同步会发现这个离线，可以进行提示），
如果重新连上，那就会拿到一个新的快照，前述逻辑就会重新成立。

### 2.3 AI助手

AI 助手在 **PC 侧** `bulletin_server.py` 后台运行（手机本机服务不跑 Ollama）。

### 已实现（当前代码）

- 多模型 `@模型名` 触发；`GET /agent` 与消息里的 `agents` / `commands` 供手机 `@`、
  `/` 补全。
- 工具 v1：附件列表/读取、`web_fetch`；Ollama tool calling 多轮。
- 内部命令 `/ai status`、`/ai stop`（不产生普通留言）；处理中 `ai_status` 状态条。
- 留言上下文过长时从**最新留言**向旧截取，**整条留言**不拆（当前为全局
  `max_board_context_chars`，待改为按模型预算，见下）。

### 目标（与角色模型对齐）

- **模型与工具按角色配置**；仅当板 `role_ids` 含触发者角色且角色允许该模型时响应。
- **上下文预算按被 @ 的模型**配置 `context_chars`，而非全局一个数适配最小窗口。
- System prompt 保持内置默认；状态行、内部命令不进 LLM 上下文。

工具分层（长期）：

1. **第一层**：列出/读取附件（只读，可不上网）。
2. **第二层**：网络查询（信息可能出网）。
3. **第三层**：受限本地命令（BASH 等，需严格目录/命令白名单）。

Agent 处理中可发 `ai_status`；同板并发 @ 排队；`/ai stop` 取消当前任务。

PC 实现说明见 [server/README.md](../server/README.md)。
