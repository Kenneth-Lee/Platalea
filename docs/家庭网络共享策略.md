# 家庭网络共享策略

## 1. 介绍

本文构想一个家庭网络共享的方案。我们使用手机，常常需要一家人（也可以理解为一个
共同的组织）一起共享一组信息。网络运营商常常在公网上提供这种服务。但很多人并不
能接受信息直接被公开到网络上。

所以，我想着用家庭（或者组织）的局域网来完成这种信息共享，基本思路是：在家庭的
PC和手机上运行一个服务，注册mDNS服务端口，这样，当我们进入局域网的时候，就通过
这个服务发现互相的服务端口，我们可以连上这些用户，然后可以给它发信息，这些信息
可以是对话，文件，目录等内容。

如果直接类比即时通讯软件，我们这个功能最难处理的是如何定义一个“用户”，因为主流
的即时通讯软件都是有一个中心的，任何一个客户端，都需要向某个中心找到自己，然后
以那个身份在不同上下文中标识自己。但我们这个方案被隔离成很多个孤岛，我们很难标
识自己。当然，一种解决思路是用一个GUID来标识用户，但为此就需要备份和迁移这个
GUID，对于我们的需求，这也创造不了什么价值。

所以放弃这种全局用户的思路。我们这样建模我们的概念空间：每个局域网中设备（手机，
PC，服务器都行），各自以自己为中心建立一个“讨论群”。进入这个局域网以后，只要对
方允许你进入这个“讨论群”，你自己给自己一个身份，在这个群里面发言，这样就可以把
天聊起来了。断开以后，这个讨论群就只留在某个用户的设备上，身份都是在这个群讨论
的时候给的。这样，这个群，本质上就是挂在你设备上的一个“留言板”，身份只属于这个
“留言板”，这就满足了我们知识“某个家庭网络”中共同通讯这个基础。

手机作为中心当然是不可靠的，因为退出这个功能，服务也不能访问了，但也足够临时使
用了。我们更长远的做法是设计一个跨平台的服务器，可以运行在Mac/Windows/Linux设备
上。你在家里开一台这样的设备，那么这个“留言板”就是永久的，你进入这个局域网，就
能随时到这个留言板上留言。它就真的成为一个家庭或者网络的“留言板”了。

然后我们把留言板的概念泛化一下，你可以创建运行多个留言板，这个留言板就变成一次
活动组织了：比如你出去旅游，可以在某个人的手机中创建一个旅行的留言板，一起去旅
游的人把日程，当天的记录，车票截图，都留言到这个留言板中。留言板就变成一个聚合
信息的地方了，你可以发言，可以发文件，发目录到这个留言板中。留言板可以导出备份，
可以导入到其他设备上，可以在不同留言板之间互相转发消息（包括附件），这样我们就
有了一个不经过任何公网服务，就在自己家的局域网里面用的数据传输服务了。

## 2. 身份控制

如前所述，我们不设计全局的用户id，所以用户的名字都是自己给的，但如果远程连入某
个留言板，我们就要控制一下你能不能维护留言板的内容了。

我们通过连入的密码来赋予身份。进入局域网打开本功能的时候，我们通过mDNS服务找到
所有支持本协议的设备。要进入设备，你要提供密码，我们按身份从高到低校验这个密码，
从而确定你的身份，这个身份决定你能看见哪些留言板，以及能在留言板上做什么事情。

我们定义如下身份：

* admin：管理员，这是最高权限，可以增删改任何留言，可以删除整个留言板，有其他任
  何用户的权限。可以导入留言板。

* user: 普通用户，这个权限，可以连入某个留言板，可以发言，可以导出整个留言板，
  但不能删改留言。

手机一侧，本地用户自然就是admin，远程连入的设备自然就是user。定义留言板的时候需
要指定哪个密码对应什么user身份，用该密码接入设备，对应用户就只有对应user的身份。
也就只对对应的留言板有user权限。

请让我们厘清这里定义的几个概念，user不是一个单独的身份，而是一类身份，它可以被
定义为“朋友”，“家人”，“同事”这种具体的身份，留言板可以赋予给不同的user身份。这些
身份并不影响该用户进入某个留言板的时候把自己叫做Tony还是Rosemary。

连入某个设备的用户只能知道密码，被连入的设备设置了密码和身份的对应关系，设备按
从高到低的顺序比较密码，赋予最高权限。连入设备的人就只看见这个密码背后的身份能
看到的留言板。就好比，有朋友到你家玩，你想让他看看你的旅行照片，你给他一个“朋友”
身份的密码，他就可以直接从你的服务器上看到“桂林旅行”这个留言板的内容。

### 3. 导入导出

这个功能主要解决在外出的问题，比如一群人出去旅行，在酒店的局域网中一起记录旅行
过程信息，回家以后想要放到家里的服务器上，那他可以在手机上导出成一个文件，然后
导入到家里的PC上就行了。

### 4. PC服务器

我们把手机的功能扩展到家庭PC上，支持Windows/Mac/Linux，这个功能更持久（手机上是
打开功能才有），随时回家都可以看到，我们尽量让这个功能变成一个无头服务器（比如
把MacMini用作无头工作站），你回家就可以随时看看上面有什么留言。所以这个服务支持
远程原理，你可以直接在手机上用管理员身份登录这些服务器，直接导入导出服务器上的
留言板。

我们在PC服务器上再集成一个AI Agent，这可以连上远程的LLM API，但对于Mac Mini这样
的环境，加上我们这个就是不想上公网而设计的，我们更多可以做一个本地的AI Agent，你
在PC服务器的留言板上@某个大模型，让它帮你总结，分析这个留言板里面的内容。就好像
这个留言板上留了一个AI管家一样。

## 5. 设计思路

### 5.1 留言板的存储管理

留言板的内容保存在应用的私有存储中（Android）或配置的 `board_root`（PC），每个留言
板一个独立目录，包括 `meta.json`、`messages.json` 与 `attachments/`（见
[家庭网络附件协议](./家庭网络附件协议.md)）。

客户端连上后同步消息列表；附件按需下载。不预设存储上限（可信环境内先用满再议）。

每个设备放一个默认留言板，便于初始化。

### 5.2 diff协议

消息序列可以是一个带序列号的单条消息的列表，如果消息是一个附件，可以带上特殊标
记说明附件的信息即可。

设备第一次连上的时候先同步一次全集，之后如果主列表发生更新，就针对这个连接加一
条“增删改”的记录，定期同步到客户端即可。

如果设备发生离线，就只能看到离线前的消息（并且同步会发现这个离线，可以进行提示），
如果重新连上，那就会拿到一个新的快照，前述逻辑就会重新成立。

### 5.3 AI助手的实现策略

AI助手只在PC服务器上实现，我们可以通过静态配置文件指定不同留言板使用的AI API和
模型参数。

留言板发言的时候可以用@model的方式触发AI进入操作。整个留言板的上下文全部进入AI
的上下文。如果留言板超过AI的允许上下文，就放弃前面的部分，AI只看到剩余的部分。

AI可以使用工具，我们按安全模式把运行的工具分三层：

* 最安全的：AI只能看到上下文，可以使用工具看附件的内容。这种情况下，只要你使用
  的是本地的AI引擎，没有任何信息泄漏到公网上。

* 只对外泄漏，不破坏本地：我们提供工具让AI上网，查看附件，这种情况下，AI只能进
  行信息处理，但它可能用它拿到的信息去做搜索，从而暴露到网络上。

* 可能破坏本地：我们提供工具让AI运行本地命令，这可能导致本地破坏。

AI和一个普通用户一样，看到你给他的信息，就去干活了，不一定会立即回复你，所以AI
在后台思考的时候，我们会在留言板视图上让你看到它的状态。我们还支持/开头的发言，
让你直接控制后台服务，比如可以用/ai status让AI立即更新状态，/ai stop停止后台工
作，等等。
