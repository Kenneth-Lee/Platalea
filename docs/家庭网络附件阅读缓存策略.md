# 家庭网络附件阅读缓存策略

## 1. 目标与范围

本策略用于规范“家庭网络 -> 留言板附件 -> 应用内直接阅读”的缓存行为，目标是：

- 尽力复用现有目录 Tab 的内置阅读能力。
- 默认提供“临时阅读，不留痕”的体验。
- 避免缓存无限增长与异常退出后的残留。
- 附件阅读流程不污染“最近打开”记录。

适用范围：

- 单文件附件阅读（首期实现）。
- 压缩阅读链路（如 `.llm.zip`、`.pic.zip`、`.md.zip`、`.html.zip`、`.epub` 等）在附件场景下的缓存管理。


## 2. 缓存分层

附件内置阅读采用双层缓存：

1. 附件原始缓存（Layer A）
   - 含义：先把留言板附件下载到应用临时目录。
   - 用途：作为后续解压/解析输入。

2. 阅读展开缓存（Layer B）
   - 含义：对需要展开的格式，从 Layer A 解压到现有阅读缓存结构。
   - 用途：供对应阅读器（md/html/epub/pic 等）直接消费。

两层缓存都必须打上来源标识：`family_attachment`。


## 3. 生命周期与删除时机

### 3.1 正常关闭阅读器

- Layer B：阅读器退出时立即删除。
- Layer A：当没有任何活跃阅读器引用该附件时立即删除。

### 3.2 退出家庭网络页面

- 强制清理所有 `family_attachment` 相关 Layer A/Layer B 缓存。
- 不依赖阅读器是否异常结束。

### 3.3 异常退出/重启兜底

- 应用下次启动时执行一次 `family_attachment` 缓存清扫。
- 以“来源标识 + 目录前缀/元数据”匹配并删除残留。


## 4. 最近打开列表策略

- 来源为 `family_attachment` 的阅读行为，禁止写入“最近打开”。
- 目录 Tab 的常规打开行为不受影响，继续按原规则写入最近记录。
- 若用户选择“下载到目录后再打开”，则按目录行为处理（可进入最近）。


## 5. 用户体验约定

- 家庭网络附件内置阅读定义为“临时阅读”。
- 需要长期保留时，用户应使用“下载到目录”路径。
- UI 文案建议明确区分：
  - 临时打开（不留最近）
  - 保存到目录（可长期管理）


## 6. 错误与可观测性

为便于排障，错误信息需包含明确原因，并区分阶段：

1. 附件拉取失败（网络、鉴权、超时、连接中断）。
2. 临时文件写入失败（权限、空间不足、URI 无效）。
3. 解压/展开失败（格式不支持、包损坏、入口缺失）。
4. 阅读器打开失败（不支持类型、初始化异常）。
5. 清理失败（文件占用、删除异常）。

日志建议至少记录：

- boardId、messageId、attachmentId
- 缓存层级（A/B）
- 文件类型与目标路径（脱敏/截断）
- 失败阶段与异常摘要


## 7. 实施顺序（最小可落地）

1. 首期：单文件附件内置阅读 + 双层缓存生命周期控制。
2. 二期：目录附件入口文件识别（`index.*`）与目录内文件点选打开。
3. 三期：更多格式与性能优化（并发下载、预热、限流）。


## 8. 验收标准

1. 在家庭网络内可直接阅读单文件附件，无需切换到目录 Tab。
2. 退出阅读器或离开家庭网络后，`family_attachment` 缓存按策略清理。
3. 附件阅读不会产生“最近打开”记录。
4. 失败时可从错误信息和日志定位到具体阶段与原因。
