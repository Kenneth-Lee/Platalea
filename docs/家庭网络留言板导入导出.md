# 家庭网络留言板：导入与导出（设计 v1）

## 1. 目标

支持「在外面用手机**本地留言板**记录 → 回家后**归档到 PC 服务器**长期维护」等场景，并与角色权限模型一致。

核心能力：

| 能力 | 说明 |
|------|------|
| **导出包** | 将一块留言板（消息 + 附件 + 元数据）打成**单个文件**，可备份、可转移 |
| **导入包** | 在**本机**或 **PC 服务器**上从包恢复为新留言板（或指定合并策略，v2） |
| **上传到服务器** | 本机导出包 → 以 admin 身份 `POST` 到服务器导入 API（等价于导入） |

现有 **`GET …/export.md`** 仅导出人类可读的 Markdown，**不含附件**；本设计的 **boardpack** 为**完整归档**，与 `messages.json` + `attachments/` 布局等价。

实现状态：设计稿（待开发）。

## 2. 使用场景

1. **旅游记录**：途中只开本机留言板服务（或不开网络服务，纯本地写）；回家后导出包，在 PC 上导入，并设置 `role_ids: ["admin", "guest"]` 供客人查看。
2. **备份**：admin 定期导出板包存到 SAF / 磁盘。
3. **迁移**：旧服务器 → 导出全部板包 → 新服务器导入。
4. **腾空间**：本机删除已上传的板；需要时再从包导入。

逻辑链：**本地板 ↔ 包文件 ↔ 服务器板**，三者通过同一格式互通。

## 3. 包格式（`.boardpack`）

采用 **ZIP** 容器，扩展名建议 `.boardpack`（MIME 可 `application/vnd.localmanager.boardpack+zip`）。

```
example.boardpack          # ZIP
  manifest.json            # 包级清单（见下）
  board/
    meta.json              # 与原 board 目录一致，含 name、role_ids（导入时可覆盖）
    messages.json
    attachments/           # 与《家庭网络附件协议》相同树形结构
      {attachment_id}/...
```

### 3.1 `manifest.json`

```json
{
  "format": "localmanager.boardpack",
  "version": 1,
  "exported_at": 1710000000000,
  "source": {
    "device": "android",
    "board_id": "uuid-on-source",
    "board_name": "2025 川藏"
  },
  "stats": {
    "message_count": 42,
    "attachment_count": 3,
    "total_bytes": 15728640
  }
}
```

- **`version`**：格式版本；导入端不兼容时拒绝并给出原因。
- **`source`**：仅供参考；导入时**生成新 `board_id`**（UUID），避免与目标环境冲突。

### 3.2 导出时 `meta.json`

- 包含 `name`、`role_ids`（若有）。
- 从**本机**导出：`role_ids` 可省略或为空，导入服务器时由 admin **强制指定**。
- 从**服务器**导出：保留当前 `role_ids`。

## 4. 操作与 API（规划）

### 4.1 本机（Android）

| 操作 | 入口建议 | 行为 |
|------|----------|------|
| 导出 | 留言板管理 · 导出包 | 写 `.boardpack` 到用户选择的 SAF 路径 |
| 导入 | 家庭网络 · 本机 · 导入包 | 解压到新 `board_id`，出现在本机板列表 |
| 删除 | 已有删除板 | 导出并确认上传后可删本机板 |

本机导入**不**经过 HTTPS API，直接写 `filesDir/family_boards/`。

### 4.2 PC 服务器

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| `GET` | `/boards/{id}/export.boardpack` | 角色可见且 admin 或 export 权限 | 流式返回 ZIP |
| `POST` | `/boards/import` | admin | `multipart` 或 `Content-Type: application/zip`，body 为包；响应 `{ board: { id, name, … } }` |

**POST import** 可选 JSON 字段：

```json
{
  "name": "覆盖显示名（可选）",
  "role_ids": ["guest"]
}
```

导入后 revision 从 0 起算，attachment_id / message_id 保持包内 UUID 或重新生成（实现时二选一并在 manifest 说明；建议**保持 UUID** 以利调试，冲突时仅新 board 目录隔离）。

### 4.3 本机 → 服务器「上传」

UI 流程：

1. 用户在本机板详情选「上传到服务器…」
2. 选择已发现的 PC 服务（或默认家庭服务器）
3. 使用 **admin 密码**（或已保存的远程 admin 凭据）连接
4. 客户端生成 boardpack（内存或临时文件）→ `POST /boards/import`
5. 可选：导入成功后提示是否删除本机板

等价于：用户手动导出再导入，一步完成。

## 5. 与 Markdown 导出的关系

| 方式 | 内容 | 用途 |
|------|------|------|
| `export.md` | 文本 + 附件文件名列表 | 阅读、打印、外发摘要 |
| `export.boardpack` | 全量可恢复 | 备份、迁移、上传服务器 |

二者并存，不互相替代。

## 6. 安全与限制

- 包内含明文消息与附件，**视为敏感**；传输走现有 TLS；文件落盘由用户负责。
- 导入仅 **admin** 可创建新板；避免客人密码向服务器注入板。
- v1 不做包内加密；若有需要可在 v2 增加可选 GPG 外层（与现有 GPG 能力对齐）。
- 超大包：导入前可检查 `stats.total_bytes`，服务器可配 `max_import_bytes`。

## 7. 实现顺序建议

1. 本机 + PC：**导出 boardpack**（只读、验证解压）
2. 本机 + PC：**导入 boardpack** → 新板
3. Android：**上传到服务器**（import API）
4. admin UI：导入时编辑 `role_ids`
5. （可选）服务器 board 列表 · 导出包
