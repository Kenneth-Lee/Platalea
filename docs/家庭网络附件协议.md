# 家庭网络留言板附件协议（v1）

## 1. 目标

在现有 HTTPS 留言板 API 上增加**大文件与目录**的收发能力，满足：

| 目标 | 说明 |
|------|------|
| 与消息分离 | `messages.json` 只保存元数据与引用；字节落在 `attachments/` |
| 可续传 | 上传、下载均支持分块与 HTTP Range |
| 目录即一项 | 用户拣选的整目录作为一条附件，内部多文件各自续传 |
| 双端一致 | Android 本机服务与 `server/bulletin_server.py` 存储布局、JSON 字段一致 |
| 本地落盘 | 下载到 SAF 根目录下固定文件夹 `留言板附件/` |

本协议为 **v1**：先实现可用路径，diff 同步、加密、配额等不在 v1 范围。

## 2. 存储布局

留言板目录在 v0 已有 `meta.json`、`messages.json`。v1 在每个 board 下增加 `attachments/`：

```
boards/{board_id}/                 # Android: filesDir/family_boards/{board_id}/
  meta.json
  messages.json
  attachments/
    {attachment_id}/
      attachment.json              # 附件元数据（见 §3.2）
      blob                         # kind=file 时的单文件内容
      files/                       # kind=directory 时
        manifest.json              # 目录内文件清单
        {file_id}/                 # 每个成员文件一个子目录（避免路径特殊字符）
          meta.json                # 相对路径、大小、sha256
          blob                     # 文件内容
```

约定：

- `attachment_id`、`file_id` 均为 UUID 字符串。
- 目录附件的 `manifest.json` 记录逻辑路径（如 `photos/a.jpg`）到 `file_id` 的映射；物理文件一律放在 `files/{file_id}/blob`，不直接 mirror 用户路径。
- 未完成上传的附件处于 `status=uploading`；`complete` 成功后为 `ready`。`failed` / `expired` 由宿主清理。

## 3. 数据模型

### 3.1 消息中的附件引用

`BulletinMessage` 增加可选字段 `attachments`（数组）。消息 JSON 示例：

```json
{
  "id": "msg-uuid",
  "seq": 12,
  "author_label": "Kenny",
  "content": "周末照片",
  "created_at": 1710000000000,
  "updated_at": 1710000000000,
  "deleted": false,
  "attachments": [
    {
      "id": "att-uuid-1",
      "kind": "file",
      "name": "readme.txt",
      "size": 4096,
      "sha256": "ab12…",
      "mime": "text/plain"
    },
    {
      "id": "att-uuid-2",
      "kind": "directory",
      "name": "周末出游",
      "file_count": 24,
      "total_size": 104857600,
      "sha256": null
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 与 `attachments/{id}/` 目录名一致 |
| `kind` | `"file"` \| `"directory"` | 单文件或目录包 |
| `name` | string | 展示名（目录为拣选时目录名） |
| `size` / `total_size` | int | 字节数；目录为成员文件之和 |
| `file_count` | int | 仅 `directory` |
| `sha256` | string? | 可选；`file` 为整文件哈希；`directory` v1 可为 null |
| `mime` | string? | 仅 `file`，可选 |

**发帖规则（v1）**

- `content` 与 `attachments` **至少一项非空**（修订现有「content 不能为空」校验）。
- 发帖时引用的每个 `attachment_id` 必须已 `status=ready` 且属于同一 `board_id`。
- 一条消息可引用多个附件；上传顺序：先传附件，再 `POST /messages` 带引用。

### 3.2 服务端 `attachment.json`

```json
{
  "id": "att-uuid-1",
  "board_id": "default",
  "kind": "file",
  "name": "readme.txt",
  "status": "ready",
  "size": 4096,
  "sha256": "ab12…",
  "mime": "text/plain",
  "chunk_size": 262144,
  "uploaded_chunks": [0, 1],
  "created_at": 1710000000000,
  "completed_at": 1710000001000,
  "uploader_device": "pixel-7-abc"
}
```

`status` 枚举：`uploading` | `ready` | `failed` | `expired`。

目录附件额外在 `files/manifest.json`：

```json
{
  "version": 1,
  "entries": [
    {
      "file_id": "f-uuid-1",
      "path": "photos/a.jpg",
      "size": 123456,
      "sha256": "cd34…"
    }
  ]
}
```

## 4. 认证与权限

与留言 API 相同，请求头：

```
X-Network-Service-Password: <角色密码>
```

**当前实现**仍为 guest / host 两档；**目标模型**见
[家庭网络留言板角色与权限](./家庭网络留言板角色与权限.md)（多角色、板级 `role_ids`）。

| 操作 | OPEN | 普通角色（板在 role_ids 内） | admin |
|------|------|------------------------------|-------|
| 初始化上传、传块、完成 | ✓ | ✓ | ✓ |
| 下载附件 / 读 meta | ✓ | ✓ | ✓ |
| 删除孤立附件（清理） | ✓ | ✗ | ✓ |
| 删除消息时附带删附件 | ✓ | ✗ | ✓ |

普通角色可上传、下载；仅 admin（或 OPEN）可删除附件与清理未完成上传。

（过渡期对照：guest ≈ 普通角色，host ≈ admin。）

## 5. HTTP API

路径前缀均为 `/boards/{board_id}`。除「传块」外，请求/响应体为 JSON；错误体与现有 API 一致：`{ "ok": false, "error": "code", "message": "…" }`。

**默认分块大小** `chunk_size = 262144`（256 KiB）。服务端可在 `init` 响应中指定；客户端必须使用返回值。

### 5.1 初始化上传（单文件）

```
POST /boards/{board_id}/attachments/init
```

请求：

```json
{
  "kind": "file",
  "name": "readme.txt",
  "size": 4096,
  "sha256": "ab12…",
  "mime": "text/plain"
}
```

响应 `200`：

```json
{
  "ok": true,
  "attachment_id": "att-uuid-1",
  "chunk_size": 262144,
  "status": "uploading"
}
```

### 5.2 初始化上传（目录）

```
POST /boards/{board_id}/attachments/init
```

请求：

```json
{
  "kind": "directory",
  "name": "周末出游",
  "entries": [
    { "path": "photos/a.jpg", "size": 123456, "sha256": "cd34…" },
    { "path": "notes.txt", "size": 100, "sha256": "ef56…" }
  ]
}
```

服务端为每个 entry 分配 `file_id`，写入 `manifest.json`，响应：

```json
{
  "ok": true,
  "attachment_id": "att-uuid-2",
  "chunk_size": 262144,
  "status": "uploading",
  "files": [
    { "file_id": "f-uuid-1", "path": "photos/a.jpg", "size": 123456 },
    { "file_id": "f-uuid-2", "path": "notes.txt", "size": 100 }
  ]
}
```

客户端按 `file_id` 分别传块（§5.3）。

### 5.3 上传分块

**单文件：**

```
PUT /boards/{board_id}/attachments/{attachment_id}/chunks/{chunk_index}
Content-Type: application/octet-stream
Body: 原始字节
```

**目录成员文件：**

```
PUT /boards/{board_id}/attachments/{attachment_id}/files/{file_id}/chunks/{chunk_index}
Content-Type: application/octet-stream
```

约定：

- `chunk_index` 从 `0` 起，覆盖 `[index * chunk_size, min((index+1)*chunk_size, size))`。
- 同一块重复 `PUT` 视为幂等（覆盖写）。
- 最后一块可短于 `chunk_size`。
- 响应 `200`：`{ "ok": true, "chunk_index": 0, "received": 262144 }`。

可选进度查询：

```
GET /boards/{board_id}/attachments/{attachment_id}
```

返回 `attachment.json` 内容及各 `file_id` 的 `uploaded_chunks`（目录）。

### 5.4 完成上传

```
POST /boards/{board_id}/attachments/{attachment_id}/complete
```

服务端校验：

- 所有必需块已到齐；
- 若提供了 `sha256`，与 `blob` 一致。

成功响应：

```json
{
  "ok": true,
  "attachment_id": "att-uuid-1",
  "status": "ready"
}
```

失败示例：`error=incomplete_upload` | `checksum_mismatch`。

未完成上传超过 **24 小时**（可配置）可被宿主清理为 `expired`。

### 5.5 发帖（扩展现有接口）

```
POST /boards/{board_id}/messages
```

请求增加可选字段：

```json
{
  "content": "周末照片",
  "author_label": "Kenny",
  "author_device": "pixel-7",
  "attachments": [
    { "id": "att-uuid-1", "kind": "file", "name": "readme.txt", "size": 4096 },
    { "id": "att-uuid-2", "kind": "directory", "name": "周末出游", "file_count": 2, "total_size": 123556 }
  ]
}
```

服务端将 `attachments` 写入消息记录；引用 id 必须 `ready`。

`GET .../messages` 快照中原样返回 `attachments` 字段。

### 5.6 下载

**读附件元数据：**

```
GET /boards/{board_id}/attachments/{attachment_id}
```

**下载单文件内容：**

```
GET /boards/{board_id}/attachments/{attachment_id}/blob
Range: bytes=0-262143
```

响应支持 `206 Partial Content`，`Accept-Ranges: bytes`，`Content-Length`，`Content-Range`。

**目录：先取 manifest，再按 file 下载：**

```
GET /boards/{board_id}/attachments/{attachment_id}/files/{file_id}/blob
Range: bytes=…
```

### 5.7 删除

```
DELETE /boards/{board_id}/attachments/{attachment_id}
```

仅宿主 / OPEN。删除消息时（`DELETE .../messages/{id}`），服务端会**一并删除**该消息引用的所有附件目录（`attachments/{attachment_id}/`），避免孤立文件堆积。

## 6. Android 客户端行为

### 6.1 上传流程

与 `docs/目录拣选模式.md` 衔接：

1. 留言板点「添加附件」→ `startDirectoryPick(BULLETIN_ATTACHMENT)`。
2. 拣选完成 → **上传队列**（前台通知 + 可取消）：
   - 对每个 `DocumentFileModel`：扫描目录条目 → `init` → 分块 `PUT` → `complete`。
3. 全部 `ready` 后：
   - 若用户填写了文字：一并 `POST /messages`；
   - 若仅附件：`content` 可为空字符串或自动摘要（如「[N 个附件]」），以 API 校验为准。
4. 回到留言板；`openBoardSession` 保持；刷新快照。

本机宿主：队列直接写 `BulletinBoardStore` + 本地 HTTPS 环回；远端走 `FamilyNetworkManager` HTTP 客户端。

### 6.2 下载流程

- 用户在消息行点附件 → 查本地是否已有缓存（见 §6.3）→ 若无则后台下载。
- 单文件：Range 续传写入临时文件，完成后 rename。
- 目录：在目标下还原 `manifest` 中的相对路径。

### 6.3 本地下载路径（SAF）

根目录下固定：

```
{rootUri}/留言板附件/{device_label}/{board_id}/{message_id}/{attachment_name}/
```

| 段 | 说明 |
|----|------|
| `device_label` | 远端设备展示名（服务名），本机留言板可用 `本机` |
| `board_id` | 留言板 id |
| `message_id` | 消息 id，便于按消息归类 |
| `attachment_name` | 引用中的 `name`；`kind=file` 时为文件名；`kind=directory` 时为目录名 |

单文件可直接落在 `…/{attachment_name}` 文件（若与目录命名冲突，加 `.file` 后缀，实现时统一规则即可）。

**索引文件（可选，v1 建议实现）**  
`留言板附件/_index.json` 记录 `attachment_id → 本地 DocumentFile URI`，避免重复下载。

### 6.4 UI（v1 最小）

- 消息列表：附件以芯片/图标展示 `name` + 大小；点击下载或打开。
- 上传中：消息可先显示「上传中」占位（客户端本地状态），成功后刷新为正式消息；或等全部完成再发帖（**默认：全部完成再发帖**，逻辑更简单）。

## 7. 错误码

| error | HTTP | 说明 |
|-------|------|------|
| `board_not_found` | 404 | 留言板不存在 |
| `attachment_not_found` | 404 | 附件 id 不存在 |
| `invalid_attachment` | 400 | 字段非法或 kind 不匹配 |
| `incomplete_upload` | 400 | complete 时块未齐 |
| `checksum_mismatch` | 400 | sha256 不符 |
| `attachment_not_ready` | 400 | 发帖时引用未 ready |
| `unauthorized` / `forbidden` | 401 / 403 | 同留言 API |

## 8. 实现顺序建议

1. **模型与存储**：`BulletinMessage.attachments`、`attachment.json`、目录 `files/` 布局（Android + Python）。
2. **单文件 API**：`init` → `chunks` → `complete` → `GET blob`（含 Range）。
3. **发帖扩展**：`POST /messages` 带引用；空 content + 附件合法化。
4. **Android 上传队列**：拣选完成 → 上传 → 发帖。
5. **目录 init + 多 file 传块**。
6. **Android 下载** → `留言板附件/` + `_index.json`。
7. **清理**：过期 `uploading`、宿主手动删附件。

## 10. 非目标（v1 不做）

- 消息 diff 同步（仍用全量快照 + revision）
- 端到端加密（依赖 HTTPS + 局域网信任）
- 附件去重（相同 sha256 跨消息共享）
- 存储配额与 LRU 淘汰
- 流式在线预览（先下载到本地再打开）
